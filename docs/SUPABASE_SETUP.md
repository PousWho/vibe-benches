# Настройка Supabase (Этап 1 — бэкенд)

Здесь по шагам: как поднять базу и подключить проект.

## 1. Создать проект в Supabase

1. Зайди на [supabase.com](https://supabase.com) и войди (или зарегистрируйся).
2. **New project** → выбери организацию, введи имя (например `vibe-benches`), пароль БД, регион.
3. Дождись создания проекта (1–2 минуты).

## 2. Создать таблицу `benches`

1. В проекте открой **SQL Editor** (слева в меню).
2. Нажми **New query**.
3. Скопируй и вставь содержимое файла `supabase/migrations/001_create_benches.sql` из этого репозитория.
4. Нажми **Run** (или Ctrl+Enter). Должно быть сообщение об успешном выполнении.

Так мы создаём таблицу и политики RLS: кто может читать и добавлять лавочки.

**Если таблица `benches` у тебя уже была создана раньше (без колонки категории):** выполни ещё раз в SQL Editor скрипт `supabase/migrations/002_add_bench_category.sql` — он добавит колонку `category`.

## 3. Взять ключи и URL

1. В проекте открой **Settings** (иконка шестерёнки) → **API**.
2. Скопируй **только**:
   - **Project URL** → это `NEXT_PUBLIC_SUPABASE_URL`
   - **anon public** (публичный ключ в разделе Project API keys) → это `NEXT_PUBLIC_SUPABASE_ANON_KEY`
   - **Не используй** ключ **service_role** / secret в приложении — в браузере будет ошибка «Forbidden use of secret API key». Он только для серверных скриптов с полным доступом.

## 4. Добавить переменные в проект

1. В корне проекта создай или открой файл `.env.local`.
2. Добавь (подставь свои значения):

```env
NEXT_PUBLIC_MAPBOX_TOKEN=твой_токен_mapbox
NEXT_PUBLIC_SUPABASE_URL=https://xxxxx.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=твой_anon_key
```

3. Сохрани файл. `.env.local` не коммитится в Git — секреты остаются локально.

## 5. Проверить

1. Перезапусти dev-сервер: `npm run dev`.
2. Открой [http://localhost:3000/map](http://localhost:3000/map).
3. Если база пустая — маркеров не будет, но надпись «Карта загружена» и отсутствие ошибки «Лавочки: …» означают, что API и Supabase работают.

### Добавить тестовую лавочку через Supabase

В **SQL Editor** выполни (подставь свои координаты при желании):

Категории: `mountain`, `forest`, `city`, `beach`, `other`.

```sql
INSERT INTO public.benches (title, description, lat, lng, category, accessibility, crowd, view, vibe)
VALUES (
  'Лавка с видом на закат',
  'Очень вайбово, особенно вечером 🌅',
  42.4309,
  19.2625,
  'mountain',
  4, 2, 5, 5
);
```

Обнови страницу карты — должен появиться маркер.

## Этап 3 — Авторизация (опционально)

Чтобы только авторизованные пользователи могли добавлять лавочки и чтобы лавочка была привязана к владельцу:

1. В Supabase включена **Authentication** (по умолчанию уже включена). В **Authentication → Providers** можно оставить Email.
2. В **SQL Editor** выполни скрипт `supabase/migrations/003_benches_user_id.sql`: добавится колонка `user_id`, политики RLS для владельца (INSERT/UPDATE/DELETE только со своим `user_id`).
3. В приложении: на карте кнопка **«Войти»**; по клику по карте без авторизации открывается модалка входа. После входа можно добавлять лавочки; в БД сохраняется твой `user_id`.

После применения 003 добавлять лавочки через SQL Editor от имени «анонима» нельзя — только через приложение под учётной записью.

**Восстановление пароля:** в приложении есть «Забыли пароль?» → вызывается API Supabase (`resetPasswordForEmail`). **Само письмо отправляет не Next.js, а проект Supabase** (встроенная почта или **SMTP**, который ты настраиваешь в **Authentication → SMTP Settings**). В коде репозитория отдельной «отправки почты» для входа нет.

- **Только в Supabase (дашборд):** **Authentication → URL Configuration → Redirect URLs** — добавь адреса, куда разрешено вести пользователя после клика по ссылке из письма (в коде для сброса пароля указан **`/auth/callback`**):
  - локально: `http://localhost:3000/auth/callback`
  - прод: `https://твой-домен.ru/auth/callback`  
  Путь **`/auth/reset-password`** в Redirect URLs **не добавляй** — на него сайт перенаправляет сам после обработки ссылки (внутренний редирект). Если в письме приходит `?code=`, его подхватывает **middleware** и меняет на сессию, затем редирект на сброс пароля.
- **В коде репозитория** браузерный клиент настроен на **implicit** для писем (не PKCE по умолчанию): в новых письмах сброса токены чаще приходят в **hash** — ссылку проще открыть из встроенного браузера почты без `access_denied`. Чтобы получить такое письмо, нужна **актуальная версия приложения** (после `git pull` / выката на хостинг) и снова «Забыли пароль». Старое письмо с `?code=` по-прежнему обрабатывает middleware (нужен тот же браузер, где запрашивали сброс).
- Если письма не доходят: проверь спам; при необходимости подключи **свой SMTP** в Supabase; смотри логи Auth. Временный обход: **Authentication → Users** → пользователь → сброс/новый пароль из Dashboard.

**Важно:** при «успешной» отправке сброса Supabase может не раскрывать, существует ли email (защита от перебора), поэтому при несуществующем адресе письма не будет, а в UI всё равно может показаться успех.

### Профили пользователей (имя, страна, дата рождения)

Чтобы при регистрации сохранять имя, страну и дату рождения и показывать их на странице профиля:

1. В **SQL Editor** выполни скрипт `supabase/migrations/004_create_profiles.sql` — создаётся таблица `profiles` и политики RLS (пользователь видит и обновляет только свою строку).
